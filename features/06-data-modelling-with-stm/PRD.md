# Data Modelling with STM: Metadata Conventions for Kimball and Data Vault

> **Status: COMPLETED** (2026-03-18). Convention dictionary, canonical examples (Kimball + Data Vault), and LLM-Guidelines.md authored for both example sets. Tooling bugs against these examples are tracked in Feature 13.
>
> **Canonical examples** are now at [`docs/data-modelling/`](../../docs/data-modelling/) with platform entry points, improved multi-source handling, and a comprehensive guide. The feature subdirectories below have been removed — see `docs/data-modelling/` for the authoritative copies.

## Goal

Define a set of **free-form metadata conventions** that allow STM to express Kimball dimensional modelling and Data Vault 2.0 patterns declaratively — without boilerplate, without grammar changes, and without losing BA readability.

The STM file stays focused on **business fields and mapping logic**. Mechanical columns (surrogate keys, hash keys, load dates, validity ranges, row hashes) are **inferred by convention** based on the metadata tokens. A BA reads the business fields. An engineer reads the tokens and knows the physical pattern. An LLM interprets both.

## Problem

Dimensional models and Data Vaults are structurally repetitive. Every SCD Type 2 dimension needs `surrogate_key`, `valid_from`, `valid_to`, `is_current`, `row_hash`. Every Data Vault hub needs `hash_key`, `load_date`, `record_source`. Every satellite needs its parent's hash key, load dates, and a hash diff.

This boilerplate:
- **Obscures business intent** — the 5 business fields drown in 6 mechanical columns
- **Invites inconsistency** — teams name and type these columns differently
- **Wastes effort** — engineers copy-paste the same pattern hundreds of times
- **Confuses BAs** — stakeholders can't tell business fields from plumbing

STM v2's `( )` metadata blocks already support arbitrary vocabulary tokens on schema blocks and fields. By defining a **standard set of conventions** for dimensional modelling and Data Vault, we get:

1. **Succinct STM files** that show only business fields
2. **Consistent mechanical columns** generated by tooling according to the conventions
3. **BA-readable specs** where the tokens communicate intent ("this is an SCD2 dimension") without requiring knowledge of the physical implementation
4. **A foundation for DDL generation**, dbt model generation, and other downstream tooling
5. **LLM-interpretable metadata** — the conventions are documented in LLM-Guidelines.md files that any AI can follow

## Approach

This is a **convention specification**, not a grammar change or a tooling feature. The metadata tokens (`hub`, `satellite`, `dimension`, `fact`, etc.) are **not reserved keywords** — they are free-form vocabulary tokens in `( )` metadata blocks, interpreted by an LLM according to documented guidelines.

The deliverables are:

1. A **convention dictionary** defining the semantics of each token
2. A set of **inference rules** documenting what fields tooling should generate for each token combination
3. **Canonical examples** (Kimball and Data Vault) modelling the same retailer domain, demonstrating the conventions in realistic, production-grade STM
4. **LLM-Guidelines.md** files alongside each example set, telling an LLM exactly how to interpret the metadata token extensions

Future work (out of scope for this feature) includes tooling that reads these conventions and generates DDL, dbt models, or other artefacts.

## Success Criteria

1. The convention dictionary covers the core patterns for both Kimball and Data Vault 2.0.
2. The canonical examples are valid STM v2 and demonstrate every convention in context.
3. A BA can read the example files and understand the business domain without knowing what "SCD Type 2" or "hub/satellite" means — the `note` blocks and comments provide that context.
4. An engineer can read the metadata tokens and unambiguously determine the full physical schema (including inferred columns) for any schema block.
5. An LLM, given the LLM-Guidelines.md, can correctly interpret the conventions and generate DDL or other artefacts.
6. The conventions do not require any changes to the STM grammar.

## Non-Goals

- **Grammar changes**: All conventions use the existing `( )` metadata syntax and `import` mechanism.
- **Tooling implementation**: No DDL generator, dbt generator, or linter is built as part of this feature.
- **Exhaustive pattern coverage**: We cover the core 80% of Kimball and DV patterns. Exotic patterns (multi-active satellites, same-as links, effectivity on links with driving keys) are acknowledged but not fully specified.
- **Runtime semantics**: The tokens describe structure and intent. Execution semantics (how a load actually runs) are out of scope.

## Schema Blocks and Roles

### The `schema` keyword

STM v2 uses a single `schema` keyword for all data structures. The role of a schema (source, target, hub, dimension, etc.) is communicated through metadata tokens in `( )`, not through different keywords. This means any schema can appear on either side of a `mapping` block — the `source { }` / `target { }` sub-blocks determine data flow direction.

### Cross-layer imports

In a layered architecture (e.g., raw vault → information mart), downstream layers consume the outputs of upstream layers. STM handles this naturally via `import`:

```stm
// mart-customer-360.stm
import { hub_customer, sat_customer_demographics, sat_customer_online } from "hub-customer.stm"

schema mart_customer_360 (dimension, scd 1, natural_key customer_id) {
  // ... mart fields ...
}

// hub_customer is defined in hub-customer.stm, but here it's on the
// source side of the mapping — and that's fine
mapping 'demographics to mart' {
  source { `sat_customer_demographics` }
  target { `mart_customer_360` }

  hub_customer.customer_id -> customer_id
  // ...
}
```

No redeclaration is needed. The imported blocks carry their metadata tokens, and tooling resolves inferred fields (hash keys, load dates, etc.) from those tokens. An imported schema with `(hub)` has its `{hub}_hk`, `load_date`, and `record_source` available for use in mappings, even though those fields are not explicitly written in the source file.

## Convention Dictionary

### Shared Conventions (Kimball + Data Vault)

| Token | Applies to | Arguments | Description |
|-------|-----------|-----------|-------------|
| `scd N` | Schema metadata | N: 1, 2, or 6 | Slowly Changing Dimension strategy. 1 = overwrite. 2 = version with history. 6 = hybrid. |
| `natural_key <field>` | Schema metadata | One or more field names | Business key used to identify a record. Same semantics as `business_key`. |
| `business_key <field>` | Schema metadata | One or more field names | Alias for `natural_key`. Preferred in Data Vault contexts. |
| `track {field, ...}` | Schema metadata | List of field names | Fields that trigger a new version when changed (SCD2/SCD6). |
| `ignore {field, ...}` | Schema metadata | List of field names | Fields that do NOT trigger a new version. |

### Kimball-Specific Conventions

| Token | Applies to | Arguments | Description |
|-------|-----------|-----------|-------------|
| `dimension` | Schema metadata | None | Marks a schema as a dimension table. |
| `conformed` | Schema metadata | None | Marks a dimension as conformed — shared across star schemas. Informational. |
| `fact` | Schema metadata | None | Marks a schema as a fact table. |
| `snapshot periodic` | Schema metadata | None | Periodic snapshot fact (semi-additive measures). |
| `snapshot accumulating` | Schema metadata | None | Accumulating snapshot fact (lifecycle milestones). |
| `grain {field, ...}` | Schema metadata | One or more field names | The grain of a fact table. |
| `ref <dim>.<field>` | Schema metadata | Dimension name + join field | Dimension reference. Tooling infers the surrogate key FK column. |
| `measure additive` | Field metadata | None | Fully additive measure. |
| `measure semi_additive` | Field metadata | None | Semi-additive measure (not summable across time). |
| `measure non_additive` | Field metadata | None | Non-additive measure (cannot be summed). |
| `degenerate` | Field metadata | None | Degenerate dimension attribute stored on the fact. |

### Data Vault 2.0 Conventions

| Token | Applies to | Arguments | Description |
|-------|-----------|-----------|-------------|
| `hub` | Schema metadata | None | Business key registry. Tooling infers: hash key, load date, record source. |
| `link` | Schema metadata | None | Relationship between hubs. Tooling infers: hash key, parent hub hash keys, load date, record source. |
| `satellite` | Schema metadata | None | Descriptive attributes (versioned). Tooling infers: parent hash key, load date, load end date, hash diff, record source. |
| `parent <hub_or_link>` | Schema metadata | Hub or link identifier | Declares which hub or link a satellite belongs to. |
| `link_hubs {hub1, hub2}` | Schema metadata | Two or more hub identifiers | Declares which hubs participate in a link. |
| `effectivity` | Schema metadata | None | Combined with `satellite`, marks an effectivity satellite. |
| `driving_key <hub>` | Schema metadata | Hub identifier | For effectivity satellites: which hub's key change drives end-dating. |

## Inference Rules

These rules define what fields **tooling should generate** for each token combination. The inferred fields are NOT written in the STM file — they are documented here and in the LLM-Guidelines.md files so that engineers and LLMs know what to expect.

### Kimball Inference

#### `dimension` (no SCD or `scd 1`)
No inferred fields. The dimension is a simple lookup table with a natural key.

#### `dimension` + `scd 2`
| Inferred field | Type | Description |
|---------------|------|-------------|
| `surrogate_key` | `BIGINT (pk, auto)` | Auto-incrementing surrogate key |
| `valid_from` | `TIMESTAMPTZ (required)` | Row effective date |
| `valid_to` | `TIMESTAMPTZ` | Row expiry date (null = current) |
| `is_current` | `BOOLEAN (required, default true)` | Current version flag |
| `row_hash` | `CHAR(64)` | Hash of tracked fields for change detection |

#### `dimension` + `scd 6`
Same as `scd 2`, plus:
| Inferred field | Type | Description |
|---------------|------|-------------|
| `current_{field}` | Same as tracked field | One "current" overlay column per `track` field |

#### `fact`
| Inferred field | Type | Description |
|---------------|------|-------------|
| `etl_batch_id` | `BIGINT` | Load batch identifier |
| `loaded_at` | `TIMESTAMPTZ` | Row load timestamp |

#### `fact` with `ref <dim>.<field>`
| Inferred field | Type | Description |
|---------------|------|-------------|
| `{dim}_key` | `BIGINT (ref {dim}.surrogate_key)` | Surrogate key FK to the dimension (one per `ref`) |

### Data Vault 2.0 Inference

#### `hub`
| Inferred field | Type | Description |
|---------------|------|-------------|
| `{hub}_hk` | `CHAR(64) (pk)` | MD5/SHA hash of business key(s) |
| `load_date` | `TIMESTAMPTZ (required)` | First-seen timestamp |
| `record_source` | `VARCHAR(100) (required)` | Source system identifier |

#### `link`
| Inferred field | Type | Description |
|---------------|------|-------------|
| `{link}_hk` | `CHAR(64) (pk)` | MD5/SHA hash of participating hub keys |
| `{hub}_hk` | `CHAR(64) (ref {hub})` | FK to each participating hub (one per `link_hubs` entry) |
| `load_date` | `TIMESTAMPTZ (required)` | First-seen timestamp |
| `record_source` | `VARCHAR(100) (required)` | Source system identifier |

#### `satellite`
| Inferred field | Type | Description |
|---------------|------|-------------|
| `{parent}_hk` | `CHAR(64) (pk, ref {parent})` | FK to parent hub or link hash key |
| `load_date` | `TIMESTAMPTZ (pk)` | Satellite version timestamp (part of composite PK) |
| `load_end_date` | `TIMESTAMPTZ` | End-of-validity (null = current) |
| `hash_diff` | `CHAR(64)` | Hash of all descriptive fields for change detection |
| `record_source` | `VARCHAR(100) (required)` | Source system identifier |

#### `satellite` + `effectivity`
Same as satellite, plus:
| Inferred field | Type | Description |
|---------------|------|-------------|
| `start_date` | `TIMESTAMPTZ (required)` | Relationship effective date |
| `end_date` | `TIMESTAMPTZ` | Relationship expiry date (null = current) |
| `is_current` | `BOOLEAN (required, default true)` | Current relationship flag |

## Risks

### Convention drift

Without grammar enforcement, teams may invent their own token variations (`dim` vs `dimension`, `bk` vs `business_key`).

**Mitigation**: The LLM-Guidelines.md files serve as the canonical reference. An LLM interpreting an STM file should follow these guidelines. Future linting tools can validate token usage against the documented conventions.

### Inference ambiguity

Different organisations may want different physical implementations for the same token (e.g., `BIGINT` vs `UUID` surrogate keys, MD5 vs SHA-256 hash keys).

**Mitigation**: The inference rules documented here are sensible defaults. Future tooling should support a configuration file to override defaults. The STM file itself is agnostic — it describes intent, not implementation.

### Token-itis: using tokens where notes suffice

Engineers might be tempted to add metadata tokens for everything rather than using `note` for non-structural information.

**Mitigation**: The design principle is clear: metadata tokens are for **structural intent that tooling or an LLM acts on**. Anything purely informational belongs in `note` blocks or `//` comments.

### Semantic validation

The grammar validates STM syntax but has no awareness of token semantics. A file with `(hub, dimension)` (contradictory tokens) is syntactically valid but semantically wrong.

**Mitigation**: Semantic validation is a linting concern, not a grammar concern. The LLM-Guidelines.md documents valid combinations. Future work: an `stm lint` tool.

## Implementation Plan

### Phase 1: Convention specification + examples (this feature)

1. Finalise the convention dictionary and inference rules (this PRD).
2. Write the canonical Kimball examples (`example_kimball/`).
3. Write the canonical Data Vault examples (`example_datavault/`).
4. Write READMEs and LLM-Guidelines.md for both example sets.

### Phase 2: Linting rules (future feature)

Build `stm lint` rules that validate:
- Token combinations are valid (e.g., `hub` and `dimension` are mutually exclusive)
- Required companion tokens are present (e.g., `satellite` requires `parent`)
- `ref` targets reference declared schemas with `dimension` metadata
- `grain` fields exist in the schema
- `track`/`ignore` fields exist in the schema

### Phase 3: DDL generation (future feature)

Build a generator that reads token-annotated STM and produces:
- DDL for the target platform (Snowflake, PostgreSQL, BigQuery, Databricks)
- dbt model stubs with appropriate materialisation strategies
- Documentation (auto-generated ERD from the tokens)
