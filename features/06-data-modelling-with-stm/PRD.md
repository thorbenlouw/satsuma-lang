# Data Modelling with STM: Tag Conventions for Kimball and Data Vault

## Goal

Formalise a set of `@tag` conventions that allow STM to express Kimball dimensional modelling and Data Vault 2.0 patterns declaratively — without boilerplate, without grammar changes, and without losing BA readability.

The STM file stays focused on **business fields and mapping logic**. Mechanical columns (surrogate keys, hash keys, load dates, validity ranges, row hashes) are **inferred by tooling** based on the tags. A BA reads the business fields. An engineer reads the tags and knows the physical pattern. Nobody writes boilerplate.

## Problem

Dimensional models and Data Vaults are structurally repetitive. Every SCD Type 2 dimension needs `surrogate_key`, `valid_from`, `valid_to`, `is_current`, `row_hash`. Every Data Vault hub needs `hash_key`, `load_date`, `record_source`. Every satellite needs its parent's hash key, load dates, and a hash diff.

This boilerplate:
- **Obscures business intent** — the 5 business fields drown in 6 mechanical columns
- **Invites inconsistency** — teams name and type these columns differently
- **Wastes effort** — engineers copy-paste the same pattern hundreds of times
- **Confuses BAs** — stakeholders can't tell business fields from plumbing

STM's `@tag()` annotation syntax already supports arbitrary metadata on schema blocks and fields. By defining a **standard tag dictionary** for dimensional modelling and Data Vault, we get:

1. **Succinct STM files** that show only business fields
2. **Consistent mechanical columns** generated by tooling according to the tags
3. **BA-readable specs** where the tags communicate intent ("this is an SCD2 dimension") without requiring knowledge of the physical implementation
4. **A foundation for DDL generation**, dbt model generation, and other downstream tooling

## Approach

This is a **convention specification**, not a grammar change or a tooling feature. The deliverables are:

1. A **tag dictionary** defining the semantics of each tag
2. A **set of inference rules** documenting what fields tooling should generate for each tag combination
3. **Canonical examples** (Kimball and Data Vault) modelling the same retailer domain, demonstrating the conventions in realistic, production-grade STM

Future work (out of scope for this feature) includes tooling that reads these tags and generates DDL, dbt models, or other artefacts.

## Success Criteria

1. The tag dictionary covers the core patterns for both Kimball and Data Vault 2.0.
2. The canonical examples are valid STM (parseable by tree-sitter) and demonstrate every tag in context.
3. A BA can read the example files and understand the business domain without knowing what "SCD Type 2" or "hub/satellite" means — the `note` blocks and comments provide that context.
4. An engineer can read the tags and unambiguously determine the full physical schema (including inferred columns) for any target block.
5. The conventions do not require any changes to the STM grammar.

## Non-Goals

- **Grammar changes**: All conventions use the existing `@tag()` annotation syntax.
- **Tooling implementation**: No DDL generator, dbt generator, or linter is built as part of this feature.
- **Exhaustive pattern coverage**: We cover the core 80% of Kimball and DV patterns. Exotic patterns (multi-active satellites, same-as links, effectivity on links with driving keys) are acknowledged but not fully specified.
- **Runtime semantics**: The tags describe structure and intent. Execution semantics (how a load actually runs) are out of scope.

## Tag Dictionary

### Shared Tags (Kimball + Data Vault)

| Tag | Applies to | Arguments | Description |
|-----|-----------|-----------|-------------|
| `@scd(type: N)` | Schema block | `type`: 1, 2, or 6 | Slowly Changing Dimension strategy. Type 1 = overwrite. Type 2 = version with history. Type 6 = hybrid (current overlay on Type 2). |
| `@natural_key(field)` | Schema block | One or more field names | The business key used to identify a record. For SCD2, this is the durable key across versions. Same semantics as `@business_key`. |
| `@business_key(field)` | Schema block | One or more field names | Alias for `@natural_key`. Preferred in Data Vault contexts. |
| `@track(field, ...)` | Schema block | List of field names | Fields that trigger a new version when changed (SCD2/SCD6). If omitted, all non-key fields are tracked. |
| `@ignore(field, ...)` | Schema block | List of field names | Fields that do NOT trigger a new version, even if they change. |
| `@source_system(field)` | Schema block | Field name | The field that records which source system provided the record. Inferred as `record_source VARCHAR(100)` if not present. |

### Kimball-Specific Tags

| Tag | Applies to | Arguments | Description |
|-----|-----------|-----------|-------------|
| `@dimension` | Schema block | None | Marks a target as a dimension table. Tooling infers a surrogate key if `@scd(type: 2)` or `@scd(type: 6)` is present. |
| `@conformed` | Schema block | None | Marks a dimension as conformed — shared across multiple star schemas. Informational; no structural inference. |
| `@fact` | Schema block | None | Marks a target as a fact table. |
| `@snapshot(periodic)` | Schema block | `periodic` or `accumulating` | Fact table snapshot type. Periodic = full state at regular intervals. Accumulating = tracks lifecycle milestones. |
| `@grain(field, ...)` | Schema block | One or more field names | The grain of a fact table — the combination of fields that uniquely identifies a row. |
| `@ref dim on field` | Schema block (inside body) | Dimension name + join field | Declares a dimension reference (foreign key). Tooling infers the surrogate key FK column. |
| `@measure(type)` | Field | `additive`, `semi_additive`, or `non_additive` | Classifies a fact measure. Additive = can be summed across all dimensions. Semi-additive = summable across some (e.g., balances across time). Non-additive = cannot be summed (e.g., ratios). |
| `@degenerate` | Field | None | Marks a dimension attribute stored directly on the fact table (no separate dimension). |

### Data Vault 2.0 Tags

| Tag | Applies to | Arguments | Description |
|-----|-----------|-----------|-------------|
| `@hub` | Schema block | None | Marks a target as a hub (business key registry). Tooling infers: hash key, load date, record source. |
| `@link` | Schema block | None | Marks a target as a link (relationship). Tooling infers: hash key, parent hub hash keys, load date, record source. |
| `@satellite` | Schema block | None | Marks a target as a satellite (descriptive attributes). Tooling infers: parent hash key, load date, load end date, hash diff, record source. |
| `@parent(hub_or_link)` | Schema block | Hub or link identifier | Declares which hub or link a satellite belongs to. |
| `@link(hub1, hub2, ...)` | Schema block | Two or more hub identifiers | Declares which hubs participate in a link. |
| `@effectivity` | Schema block | None | Combined with `@satellite`, marks an effectivity satellite that tracks the temporal validity of a link relationship. Tooling infers: start/end dates, is_current flag. |
| `@driving_key(hub)` | Schema block | Hub identifier | For effectivity satellites: which hub's key change drives end-dating. |

## Inference Rules

These rules define what fields **tooling should generate** for each tag combination. The inferred fields are NOT written in the STM file — they are documented here and in the example READMEs so that engineers know what to expect.

### Kimball Inference

#### `@dimension` (no SCD or SCD Type 1)
No inferred fields. The dimension is a simple lookup table with a natural key.

#### `@dimension @scd(type: 2)`
| Inferred field | Type | Description |
|---------------|------|-------------|
| `surrogate_key` | `BIGINT [pk, auto]` | Auto-incrementing surrogate key |
| `valid_from` | `TIMESTAMPTZ [required]` | Row effective date |
| `valid_to` | `TIMESTAMPTZ` | Row expiry date (null = current) |
| `is_current` | `BOOLEAN [required, default: true]` | Current version flag |
| `row_hash` | `CHAR(64)` | Hash of tracked fields for change detection |

#### `@dimension @scd(type: 6)`
Same as Type 2, plus:
| Inferred field | Type | Description |
|---------------|------|-------------|
| `current_{field}` | Same as tracked field | One "current" overlay column per `@track` field |

#### `@fact`
| Inferred field | Type | Description |
|---------------|------|-------------|
| `etl_batch_id` | `BIGINT` | Load batch identifier |
| `loaded_at` | `TIMESTAMPTZ` | Row load timestamp |

#### `@fact` with `@ref dim on field`
| Inferred field | Type | Description |
|---------------|------|-------------|
| `{dim}_key` | `BIGINT [ref: {dim}.surrogate_key]` | Surrogate key FK to the dimension (one per `@ref`) |

### Data Vault 2.0 Inference

#### `@hub`
| Inferred field | Type | Description |
|---------------|------|-------------|
| `{hub}_hk` | `CHAR(64) [pk]` | MD5/SHA hash of business key(s) |
| `load_date` | `TIMESTAMPTZ [required]` | First-seen timestamp |
| `record_source` | `VARCHAR(100) [required]` | Source system identifier |

#### `@link`
| Inferred field | Type | Description |
|---------------|------|-------------|
| `{link}_hk` | `CHAR(64) [pk]` | MD5/SHA hash of participating hub keys |
| `{hub}_hk` | `CHAR(64) [ref: {hub}]` | FK to each participating hub (one per `@link` argument) |
| `load_date` | `TIMESTAMPTZ [required]` | First-seen timestamp |
| `record_source` | `VARCHAR(100) [required]` | Source system identifier |

#### `@satellite`
| Inferred field | Type | Description |
|---------------|------|-------------|
| `{parent}_hk` | `CHAR(64) [pk, ref: {parent}]` | FK to parent hub or link hash key |
| `load_date` | `TIMESTAMPTZ [pk]` | Satellite version timestamp (part of composite PK) |
| `load_end_date` | `TIMESTAMPTZ` | End-of-validity (null = current) |
| `hash_diff` | `CHAR(64)` | Hash of all descriptive fields for change detection |
| `record_source` | `VARCHAR(100) [required]` | Source system identifier |

#### `@satellite @effectivity`
Same as satellite, plus:
| Inferred field | Type | Description |
|---------------|------|-------------|
| `start_date` | `TIMESTAMPTZ [required]` | Relationship effective date |
| `end_date` | `TIMESTAMPTZ` | Relationship expiry date (null = current) |
| `is_current` | `BOOLEAN [required, default: true]` | Current relationship flag |

## Example Domain: RetailCo International

Both the Kimball and Data Vault examples model the same fictional retailer to enable direct comparison.

### Company Profile

**RetailCo International** is a multinational department store chain operating 850+ stores across 12 countries. They sell apparel, home goods, electronics, and beauty products through physical stores and an e-commerce platform.

### Source Systems

| System | Technology | Description |
|--------|-----------|-------------|
| `pos_oracle` | Oracle Retail POS | In-store transactions from 850+ registers. High volume (~2M transactions/day). Legacy field naming. |
| `ecom_shopify` | Shopify Plus | Online orders. Modern JSON-based schema. ~200K orders/day. |
| `loyalty_sfdc` | Salesforce Service Cloud | Customer profiles and loyalty programme. 18M members. |
| `merch_sap` | SAP MM | Product master data, categories, suppliers, pricing. ~500K active SKUs. |
| `wms_manhattan` | Manhattan Associates WMS | Warehouse inventory across 24 distribution centres. Real-time stock positions. |

### Domain Coverage

Both examples cover four business domains:

| Domain | Key entities | Sources |
|--------|-------------|---------|
| **Customer** | Customer identity, demographics, loyalty | loyalty_sfdc, pos_oracle, ecom_shopify |
| **Product** | Product master, categories, brands, suppliers | merch_sap |
| **Store** | Physical stores, regions, formats | pos_oracle (enriched with reference data) |
| **Sales** | Transactions, line items, revenue | pos_oracle, ecom_shopify |
| **Inventory** | Stock positions, warehouse locations | wms_manhattan |

## Risks

### Tag proliferation

Too many tags makes the convention hard to learn and remember.

**Mitigation**: The dictionary is deliberately small — 19 tags total across shared, Kimball, and Data Vault. Most files use 4–6 tags. The tags are named to match established terminology (any data engineer recognises `@hub`, `@dimension`, `@scd`).

### Inference ambiguity

Different organisations may want different physical implementations for the same tag (e.g., `BIGINT` vs `UUID` surrogate keys, MD5 vs SHA-256 hash keys).

**Mitigation**: The inference rules documented here are sensible defaults. Future tooling should support a configuration file to override defaults (hash algorithm, surrogate key type, column naming conventions). The STM file itself is agnostic — it describes intent, not implementation.

### Tag-itis: using tags where notes suffice

Engineers might be tempted to tag everything rather than using `note '''...'''` for non-structural metadata.

**Mitigation**: The design principle is clear: tags are for **structural intent that tooling acts on**. Anything purely informational belongs in `note` blocks or comments. The tag dictionary is the definitive list — if a tag isn't in the dictionary, it shouldn't be used for modelling.

### Grammar validation

The tree-sitter parser validates STM syntax but has no awareness of tag semantics. A file with `@hub @dimension` (contradictory tags) is syntactically valid but semantically wrong.

**Mitigation**: Semantic validation is a linting concern, not a grammar concern. Future work: an `stm lint` tool that validates tag combinations against the dictionary. For now, the examples and documentation serve as the reference.

## Implementation Plan

### Phase 1: Convention specification + examples (this feature)

1. Finalise the tag dictionary and inference rules (this PRD).
2. Write the canonical Kimball examples (`example_kimball/`).
3. Write the canonical Data Vault examples (`example_datavault/`).
4. Write READMEs for both example sets.
5. Validate all `.stm` files parse correctly with tree-sitter.

### Phase 2: Blue-sky document update (this feature)

Update `IDEAS-BLUESKYTHINKING.md` to reference this PRD as the formalised version of sections 4, 5, and 6.

### Phase 3: Linting rules (future feature)

Build `stm lint` rules that validate:
- Tag combinations are valid (e.g., `@hub` and `@dimension` are mutually exclusive)
- Required companion tags are present (e.g., `@satellite` requires `@parent`)
- `@ref` targets reference declared dimensions
- `@grain` fields exist in the schema
- `@track`/`@ignore` fields exist in the schema

### Phase 4: DDL generation (future feature)

Build a generator that reads tagged STM and produces:
- DDL for the target platform (Snowflake, PostgreSQL, BigQuery, Databricks)
- dbt model stubs with appropriate materialisation strategies
- Documentation (auto-generated ERD from the tags)
