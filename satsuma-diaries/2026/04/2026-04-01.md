# The Satsuma Diaries
## Wednesday, 01 April 2026

> **tl;dr** — An absolutely mental day: a new browser-based tool for visualising data pipelines shipped with automated tests, a shared code library was extracted so multiple tools stop duplicating work, and somewhere around twenty-three separate bugs got fixed across the CLI, the language server, and the visualisation layer. Forty-four commits. On a Wednesday.

Forty-four.

I'm not going to make a joke about it being April Fools' Day. I'm going to let the number speak for itself. Forty-four commits. Eleven pull requests merged. I had to make a spreadsheet just to keep track.

Right, so the headline is **Feature 29** landing — and it's a properly big one. The idea is that the bit of code responsible for building the visual representation of your data pipeline (what schemas exist, how data flows between them, which fields map to which) was buried inside the VS Code extension's language server. That's fine if you only ever use VS Code, but it means nothing else can use it without going through VS Code, which is mad. So the team extracted it into its own shared package, `@satsuma/viz-backend`, and then built a standalone browser harness — basically a lightweight web app that lets you load a Satsuma file and render the pipeline diagram without needing VS Code at all.

The harness isn't just for humans. It exposes automation hooks so that Playwright — a browser testing framework — can drive it headlessly and make assertions. Eight browser tests shipped: overview rendering, drilling into mapping detail, hovering over fields to see connections, navigating across files with cross-file lineage, layout stability. That last one sounds boring but "does it actually look right" is the kind of thing that breaks silently, so good. All eight passing.

Getting there took a few goes. There was a VS Code bundling issue where the extension couldn't find the viz component because the release build script hadn't been updated to build `satsuma-viz` first. Fixed. There was a CodeQL static analysis warning about using `innerHTML` — a security scanner flag for "you're putting HTML directly into the page, are you sure?" — which got suppressed in the right place. Fixed. CI caching for the new shared backend package. Fixed. Playwright apparently can't run headlessly on ARM macOS without crashing (something about a graphics driver), so the team built a sentinel-file protocol: the human starts a watcher script once, and the agent drops a file called `.run-tests` to trigger a run, then picks up the results from `.playwright-results.txt`. Bit of a palaver but it works.

While all that was going on, someone also fixed **two separate bugs in the LSP** — the language server that powers VS Code autocomplete and error highlighting. One was that quoted join descriptions inside `source {}` blocks were being indexed as actual schema references, triggering false "undefined source" warnings. A join description is just a human-readable label, not a real schema name — the language server wasn't distinguishing them. The other was nested field coverage: if your mapping said `customer.email -> customer_email`, the source schema card in the VS Code panel would show `customer.email` as unused even though it was being mapped. Both fixed.

Then there's the bugs. Not a few bugs. A *lot* of bugs. The validate and lint commands (which check your Satsuma files for problems) had six separate issues: anonymous mappings — ones without a name — were being silently skipped during fix operations; error messages for certain references were showing internal key names like `note:metric:revenue` instead of just `revenue`; two commands were using different rule names for the same check, making deduplication impossible; and two more edge cases around anonymous mappings in namespaced files and `@ref` links in file-level notes. All six fixed, all with regression tests.

The `diff` command — which shows you what changed between two versions of your data pipeline — had its own pile: schema note blocks being misread as field changes, missing documentation for a JSON key, and a dispatch table refactor replacing a fourteen-branch if-else chain with something a human can actually read. The `diff` command's inner loop also had an O(n²) performance issue — meaning as your pipeline got bigger, it got exponentially slower rather than linearly slower. That got fixed too.

Namespace handling in the output commands had six bugs, mostly around the CLI echoing back raw query strings instead of canonical schema names. So you'd query for `customers` and get `customers` in the output, but the canonical name was `crm::customers`. Not great for downstream tooling.

There were four arrow/NL bugs (the "arrows" command shows data flow, and NL refs are the plain-English annotations you write in mapping files pointing to source fields). Anonymous mappings inside namespace blocks weren't being indexed with the namespace prefix. nl-derived arrows — the ones computed from those plain-English annotations — weren't showing up when you queried by target field. Fixed.

Four more quick fixes: a missing JSON key in `find` output, the `lineage` command returning plain-text errors instead of JSON when in `--json` mode, `lint` incorrectly returning an error exit code for warning-only findings, and `summary` miscounting total errors.

Somewhere in all of this there was also a TypeScript migration — all 26 test files in the CLI package were converted from JavaScript to TypeScript, gaining type checking. And CI coverage measurement was added: the test suite now reports line coverage with a 70% threshold. Current coverage is 88.4%. That's... actually respectable.

There's also an AGENTS.md update explaining to future AI agents exactly how the Playwright sentinel-file protocol works, because apparently agents keep trying to run Playwright directly (it crashes) or don't know the protocol exists. Documenting it in the one place every agent reads first is the sensible call.

On God, I'm tired just reading this.
