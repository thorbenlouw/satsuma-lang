# The Satsuma Diaries
## Thursday, 02 April 2026

> **tl;dr** — The word "metric" has been removed from the Satsuma language. Not the concept — metrics still exist — but the special keyword. You now declare a metric the same way you declare everything else, just with a tag that says it's a metric. An architectural decision record was written. Six commits.

Sometimes a change is small on the outside and surprisingly large on the inside.

The `metric` keyword in Satsuma used to be its own special thing — you'd write `metric monthly_recurring_revenue` and get a dedicated block type with its own syntax for declaring grain, source, and value. It was different from `schema`, different from `mapping`. A first-class citizen.

As of today, it's gone. The language no longer has a `metric` keyword. What you write now is a `schema` block with `(metric)` in its metadata parentheses — the same decoration syntax that `report` and `ml_model` already use. The data-flow relationship to metrics is now expressed through an explicit `mapping` block, just like every other source-to-target connection in the language.

PR #191 explains the why pretty plainly: the first-class metric syntax was creating two parallel systems for what is fundamentally the same thing. A metric schema is still a schema. Data flowing *into* it is still a mapping. Treating it differently at the syntax level was adding complexity without adding expressiveness. The team wrote ADR-024 — an Architectural Decision Record, basically a permanent note saying "here is the decision, here is why, and here is what it supersedes" — to document the call. ADR bodies don't get edited after the fact, so if this turns out to be wrong in six months, a new ADR gets written; this one stays.

In terms of blast radius: the grammar was updated, the parser regenerated, the corpus tests (the automated fixture tests that prove the parser produces the right tree for given input) updated, and every layer that dealt with metrics — the CLI `metric` command, the language server hover/symbols/code lens/folding, the viz backend extraction, the spec, the site — all updated. The full test suite is green.

PR #192 cleaned up the aftermath: one open ticket (`sl-vces`, about a `sources{}` structural block for metrics) was closed as will-not-fix because it was designed around the old keyword that no longer exists. Two empty feature stub directories got removed.

Also today: three more JSON output bugs in the `lineage` and `graph` commands got fixed and a PR raised — depth-boundary dangling nodes, inconsistent edge key names between commands, and orphan nodes in cross-namespace graph output. That's PR #193, just opened, not yet merged. Pip is watching.

Six commits total. Quiet by the standards of yesterday. But removing a keyword from a programming language and updating every single thing that touched it is not nothing.
