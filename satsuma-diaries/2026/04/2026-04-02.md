## Thursday, 02 April 2026

> **tl;dr** — Satsuma 0.7.0 shipped. The `metric` keyword was removed from the language entirely, every pipe step is now natural language, the import system got proper symbol-level scoping, and the whole thing was tagged and released. Ten pull requests. Twenty-nine commits. PR #200.

Zero point seven point zero.

I'm going to let that breathe for a second because yesterday was forty-four commits and today is another twenty-nine and at some point someone has to say: bruv. Bruv. Take a walk. Touch some grass. Do literally anything that doesn't involve a tree-sitter grammar.

Right, so the big one. **The `metric` keyword is dead.** Gone. Removed from the language. PR #191 is the one that does it and I need to explain what happened because it's actually a proper design decision, not just tidying up.

Satsuma used to have this dedicated keyword — `metric` — for declaring things like monthly recurring revenue or churn rate. You'd write `metric monthly_recurring_revenue` and get a special block type with its own syntax for grain, source, value, the lot. Separate from `schema`, separate from `mapping`. Its own whole thing.

The problem, as the architectural decision record (ADR-024 — basically a permanent note explaining a big design call and why it was made) puts it quite plainly: a metric schema is still a schema. Data flowing into it is still a mapping. Having a separate keyword created two parallel systems for what is fundamentally the same operation. So now a metric is just a `schema` block with `(metric)` written in its metadata parentheses — the same decoration syntax that `report` and `ml_model` already use. You want to express how data flows into your metric? Write a `mapping` block, same as everything else.

The blast radius of removing a keyword from a programming language is, as it turns out, enormous. Grammar updated. Parser regenerated. Corpus tests updated. Core extraction and validation and formatting updated. The CLI's `metric` command updated. The language server's hover, symbols, code lens, and folding — all updated. The viz backend extraction layer. The spec. The site. The examples. Every single layer that knew the word "metric" had to learn to forget it. All tests green at the end. 862 CLI tests, 297 core tests, 216 LSP tests, 84 viz-backend tests. Not a single one left behind.

Then, while the metric keyword's body was still warm, **Feature 28 landed** — pipeline simplification (PR #196). This one's a bit more philosophical. Satsuma used to classify pipe steps — the individual operations in a data transformation chain — as "structural" (things like `{ * 100 }` that look like code), "NL" (natural language descriptions like `"Normalize to monthly"`), or "mixed" (both). As of today, that entire axis is gone. Every pipe step is now implicitly a natural language string. The grammar no longer even has rules for arithmetic operators or function-call syntax inside pipes. Nine tickets, eight implementation PRs covering grammar, core, CLI, LSP, viz model, viz rendering, VS Code extension, spec, docs, and examples. All merged.

The reasoning, for anyone keeping score: the structural classification was adding complexity without adding value. Nobody was writing `{ round(2) }` in their Satsuma files. The NL descriptions are what humans and AI agents actually read. The structural syntax was a vestige of an earlier vision where Satsuma might contain executable transform logic, and that ship sailed a while ago. Removing it makes the language simpler, makes the tooling simpler, and makes the visualisation layer simpler — no more orange edges for "pipeline" transforms, no more classification filter dropdown in VS Code. Just text.

Between the metric removal and the pipeline simplification, today was essentially "make the language smaller and more coherent" day. That's a rare thing. Most days you add stuff. Today, stuff got taken away, and the thing that remained was better for it.

In among all of that, two properly solid bug fixes landed. PR #197 found that the `summary` command was overcounting something called nl-derived edges — when an `@ref` annotation (a plain-English note pointing to a source field) resolved to a field that was *already* explicitly declared as a source, it was being counted as a bonus edge. The `graph` command had dedup logic for this; `summary` didn't. Now they agree. And PR #198 added **symbol-level import reachability** to the core library — previously, if you imported a file, you got access to everything in it. Now the system actually traces which specific symbols (schemas, mappings) are reachable through your import chain. If you reference something that exists in the workspace but isn't reachable from your imports, you get an error. This is the kind of thing that catches real bugs in large multi-file platforms where you think you've imported something but you haven't.

Three more JSON output bugs got squashed in PR #193: the `lineage` command was emitting dangling mapping nodes at depth boundaries, using different key names for edges than `graph` (one said `src/tgt`, the other said `from/to`), and cross-namespace graph output was referencing nodes that didn't exist in the nodes array. All three now have regression tests.

And then, right at the end of the day: PR #199 shipped `@ref` highlighting in the viz layer. Those `@ref` annotations — `@status`, `@arr_value`, the little plain-English breadcrumbs that tell you where a field's data comes from — now render in bold dark green in the mapping detail view and edge layer transform cards. Small thing visually. But it means when you're reading a complex transformation and trying to follow the data trail, the references to source fields actually stand out from the surrounding prose instead of blending in. Also fixed a flaky Playwright test and hooked the viz harness into the root install scripts, which is the kind of thing that breaks fresh clones silently.

PR #200 wrapped it all up. Version bumps across CLI, LSP, and VS Code extension. A changelog covering everything since v0.6.0. Feature 28. Feature 29. Forty-eight bugs filed and fixed. Fourteen retrospective ADRs. Symbol-level import reachability. File-based CLI entry points. Core extraction consolidation. Tagged and released.
