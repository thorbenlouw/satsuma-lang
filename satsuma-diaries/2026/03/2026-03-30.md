# The Satsuma Diaries
## Sunday, 30 March 2026

> **tl;dr** — An absolute Sunday bloodbath of constructive refactoring: all duplicated code across the CLI, LSP, and visualiser was moved into a single shared library; the language server was extracted so non-VS-Code editors can use it; ten architectural decisions were formally recorded; and every completed feature was archived. Sixty-five commits. On a Sunday.

Right.

Sixty-five commits. Twenty pull requests merged. I have to break this into sections or I will lose my mind.

**The consolidation.** The project had a problem: three different tools — the CLI, the VS Code language server (the thing that gives you red squiggles and autocomplete), and the visualiser — all needed to understand the same Satsuma files. Each had built its own version of that understanding. Separate implementations of the same logic, drifting apart over time, each one a future bug waiting to happen. PR #135 opened the floodgates: five ticket's worth of shared logic moved into `@satsuma/core`. String utilities, the parser singleton, error node collection, mapping coverage types, and a whole new `@satsuma/viz-model` package for the data contract the visualiser uses. Then a second wave across six more PRs extracted the rest of what the language server had been hoarding: CST text helpers, field extraction, metadata extraction, classification, spread resolution, semantic diagnostics. Twenty-six new tests added during the migration. A regression in `sourceRefNameNs` — the function that handles namespace-qualified source references — was caught and fixed along the way. The epic `sl-jvwu` is now closed.

To put that in normal words: when you have three tools that all need to do the same thing, you write it once and have all three call it. Less code, fewer bugs, one place to fix things when they go wrong. The kind of refactoring that doesn't look exciting in a demo but saves six months of pain later.

**The LSP extraction.** Alongside the consolidation, the language server got its own package (PR #154). Previously it lived inside the VS Code extension folder — fine for VS Code users, useless for everyone else. It's now at `tooling/satsuma-lsp/` with its own `bin` entry. Meaning: `npx satsuma-lsp --stdio` and any editor that speaks LSP — Neovim, Helix, Emacs, whatever — gets the full experience. Red squiggles, go-to-definition, completions. Two hundred and ninety-three tests came along. That's ADR-021 implemented.

**The ten ADRs.** An ADR is an Architectural Decision Record — a permanent note that says "this is what we decided, this is why, and this is what we didn't do." PR #149 landed ten of them retroactively: the LSP server architecture, the eager workspace index strategy, the choice of ELK.js for visualisation layout, `@ref` as implicit field lineage, the v1-to-v2 language migration, Eleventy as static site generator, the universal CLI build, and more. Decisions that had been made months ago and lived only in git history and someone's head, now formally documented. Then ADR-020 and ADR-021 arrived the same day (#150) — laying out the consolidation and extraction plans. Writing the architectural plan and executing it on the same Sunday. Remarkable work ethic, genuinely.

**The smoke tests.** Forty-three imperative Python test functions, most of them written ages ago, got rewritten as Gherkin scenarios using `pytest-bdd` (PR #144). Gherkin is the "Given/When/Then" format — readable by humans, automatable by test runners. Four new lineage scenarios were added that actually assert on output. The old `lineage.py` referenced a non-existent fixture and had zero assertions. I did a double-take at that. Wired into CI with JUnit reporting.

**The visualiser.** PR #148 added full transitive lineage with a file-scope filter to the mapping diagram — you can now see data flowing across multiple files and toggle down to a single file's perspective. Field notes, arrow notes, and left-aligned transform labels landed alongside it (#141 follow-up commits). Schema card labels and notes got capped at 400px so they don't take over the screen. It's... actually looking quite good now.

**The housekeeping.** The CI deploy-site workflow YAML was genuinely broken — a Python heredoc at column 1 confused the parser. Fixed, then `yamllint`, `ruff`, and `markdownlint` added to the lint pipeline (#139). Two moderate dependency vulnerabilities patched (#137). The LSP server was crashing because the bundled `web-tree-sitter` uses `import.meta.url` which esbuild mangles in CJS bundles — fixed with a dynamic import workaround and then an esbuild banner shim (#146). `clean:all` expanded to actually clean everything.

**The archiving.** PR #155 moved seven completed feature PRDs to the archive. Then five more features got individually archived: language simplification (all five phases done), the CLI WASM migration, the viz site migration, VS Code UX improvements. The `features/` directory is now empty. Every feature that was planned for the current roadmap has been built. Allow it.

**The docs.** `@ref` terminology replaced every remaining mention of "backtick ref" across twenty-two files (#134). Docs reorganised into `developer/` and `product-owner/` subfolders (#142). Literate comments added to six modules (#141). Convention rules extracted with named constants and citations (#140). Redundant "Where Used" VS Code command removed (#147).

Version bumped to v0.6.0 somewhere in the middle of all this. Pip barely had time to notice.
