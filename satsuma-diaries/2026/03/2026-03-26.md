# The Satsuma Diaries
## Thursday, 26 March 2026

> **tl;dr** — `@ref` became first-class syntax, single quotes were dropped from the grammar, the site moved to a proper static site generator, and a mapping visualisation launched in VS Code. Ten PRs, seventy-two commits.

Seventy-two commits. Thursday. Several distinct things happened and I'm going to try to give them each their due.

The grammar was simplified in three places via the language simplification phases 3 and 4 (#105). The metadata value grammar, which used to have thirteen different forms for different kinds of values, was collapsed into one greedy rule: `value_text`. The pipe step grammar — the chain of transforms you write after an arrow — became `pipe_text` instead of a complicated token-call/arithmetic-step distinction. The map entry grammar similarly got simplified to greedy matching for both key and value. "Greedy" here means the parser just takes as much text as it can up to the next delimiter, rather than trying to classify the specific type of thing it's reading. Less grammar complexity, same expressive power, better error recovery. That's solid engineering.

`@ref` landed as a first-class language feature via PR #104. An `@ref` is a cross-reference annotation you write in your Satsuma files — like `@ref crm::customers.account_id` — to explicitly say "this thing refers to that thing." Before today, you could only capture these references by embedding field names in backtick notation inside natural language strings. Now there's proper syntax for it: its own grammar rule, its own highlighting in the TextMate grammar, extracted by the NL ref extractor, participating in lineage traversal, checked by the validator, handled by the LSP server. The hidden-source-in-nl lint rule was also promoted from a warning to an error — if you're referencing a source field in a natural language description without using the proper syntax, the linter now refuses to let it through. All the example files were migrated to the new `@ref` syntax in a separate PR (#106) to keep the diff clean.

Single quotes were also dropped from the grammar entirely (#105). Quoted names in Satsuma now use only backticks. Every example file was migrated. Every test. Good — fewer ways to do the same thing means less confusion about which one to use.

Twelve more CLI bugs were found, filed (#108), and then fixed (#110) in the same day. The bugs-then-fixes pattern is becoming a habit and I respect the discipline. LSP Find All References, the coverage command, and where-used were also fixed (#111).

The site was migrated to Eleventy, a static site generator (#109). Eleventy is a tool that turns templates and content into flat HTML files that can be served cheaply and fast. It replaced the previous hand-crafted static HTML. The move means the site is now maintainable at scale — templates, layouts, data files — rather than one monolithic HTML page.

And then: the mapping visualisation. A PRD landed for it (#112) and then the actual build happened the same day (#114, #115). A new package called `@satsuma/viz` — a set of web components (reusable UI building blocks for browsers) built using Lit, a lightweight web component framework. Alongside it: ELK.js, a layout engine that figures out where to put boxes and arrows in a diagram. The VS Code panel to host the whole thing: you open a mapping file, you get a visual diagram of how the schemas connect. The boxes are schema cards, the arrows are mapping arrows, and the whole thing is interactive. The PR for the visualisation (#115) describes it as Feature 23, Phase 1 through full implementation. v0.4.0 shipped by end of day.

Seventy-two commits. I said I'd give each thing its due. I tried.
