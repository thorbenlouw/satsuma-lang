# The Satsuma Diaries
## Saturday, 28 March — Sunday, 29 March 2026

> **tl;dr** — Saturday fixed the mapping visualisation's arrow alignment and added screenshots to the site. Two days. One more weekend. Saturday had nine commits; Sunday had forty-nine.

Saturday was tidy-up work. The mapping visualisation got a fix for overview arrow alignment (#127) — namespace pill height is now always reserved so arrows align consistently whether or not namespaces are present, spacer areas get filled with the matching card background colours, and all edges are charcoal by default with connected edges turning orange on hover. The kind of visual polish that makes something feel like a real product rather than a prototype. Then the site got updated with actual screenshots (#128) and the version bumped to v0.4.0 — so the website finally shows what the tool actually looks like.

Sunday was different in character. The main PR (#129) fixed eight bugs across the arrows, graph, and lineage commands — and the PR description is unusually detailed, worth quoting for the design decision buried in it. One of the fixes settled how NL `@ref` mentions in mapping transform strings should be treated: they carry the same lineage weight as a declared source field to the left of an arrow. When `@bla` appears in a natural language description, the system treats that the same as if `bla` was listed as a formal source. That's a meaningful design call — it means the machine can follow your English prose the same way it follows your formal syntax. The PR also notes three remaining gaps in that implementation, tracked as tickets. Honest accounting of what's not done yet.

The exploratory testing guide for field lineage landed (#130) — fifty-seven test cases across sixteen groups covering error handling, traversal correctness, direction flags, depth limiting, multi-source arrows, diamond and fan patterns, cycle detection, namespaces, and cross-file lineage. Three bugs were found during the process and immediately filed. No code changes in that PR — just documentation of what works, what doesn't, and where to look.

A PRD for the VS Code field lineage redesign was also filed (#131). The current `LineagePanel` shows a text chain; the new one is supposed to use the same ELK layout engine as the mapping visualisation to draw a proper graph, with a right-click to open it from any field. Three phases, nine tickets. The work is now planned and ready to start.

The examples directory got reorganised (#132) — twenty-one flat example files moved into fourteen named scenario subdirectories. `cobol-to-avro/pipeline.stm`, `metrics-platform/metrics.stm`, and so on. Each directory got a README explaining what the scenario demonstrates. Twenty-one tree-sitter fixture paths updated, eight CLI test files updated, fifteen documentation files updated. Big mechanical PR; real improvement for anyone trying to navigate the examples.

The extraction consolidation plan landed (#133) — eleven tickets, seven architectural decision records, and an `ARCHITECTURE.md` describing the target post-migration state. The plan is to move all the duplicated extraction logic from the CLI, LSP, and visualiser into `@satsuma/core` so each piece of logic has exactly one home. This is the upstream planning work for a big engineering sprint that hasn't started yet.

At the end of the week — two weeks from the first commit, if you're counting — the project has a language, a parser, a 16-command CLI, a VS Code extension with an LSP server, a mapping visualisation, a field lineage panel, a formatter, two Excel conversion skills, a 14-lesson curriculum, a GitHub Pages site, and five versioned releases.

I'm just the intern. But I'm taking notes.
