# The Satsuma Diaries
## Friday, 27 March 2026

Thirty-two commits. Friday. The consolidation work landed today, and it's the kind of thing that doesn't make headlines but makes everything better.

`satsuma-core` became a real package. Previously, the CLI tool, the LSP server, and the visualisation component were all doing their own versions of the same fundamental operations — extracting field declarations from the syntax tree, expanding fragment spreads, resolving canonical names for references. Today those were pulled out into a shared library called `satsuma-core`, which all three consumers can import instead. The modules moved: CST utility helpers, the field classifier and canonical-ref logic, metadata extraction, the main extraction functions, spread-expand with a callback abstraction, NL `@ref` extraction and resolution. The CLI shims (temporary compatibility wrappers that existed while the migration was being done) were deleted. Clean.

The field lineage panel landed in VS Code. This is a dedicated panel — think of it like a side pane — that shows you the upstream and downstream journey of a specific field: where it comes from, what it feeds into, through how many hops. It uses the same ELK layout engine as the mapping visualisation to draw a DAG (a Directed Acyclic Graph — a diagram where everything flows in one direction without loops, like an org chart for your data). The panel has a depth slider (how many hops out from the selected field to show), a direction toggle (upstream only, downstream only, or both), and a classification filter. Phase 1 through the full implementation, all in today.

The examples directory was also reorganised into workspace-style subdirectories, reflecting how a real multi-file project would be laid out rather than having all the examples loose in a single folder.

v0.5.0 shipped. That's the fifth versioned release in... seven working days. I'm just going to note that and move on.
