# The Satsuma Diaries
## Monday, 31 March 2026

> **tl;dr** — Forty-eight bugs filed in one exploratory session. The CLI contract changed so it rejects folder paths and only accepts actual files. JSON output was standardised across seven commands. Also six dependency bumps, which is the least interesting sentence in this entry.

Right so on paper, Monday after a sixty-five-commit Sunday should be a recovery day. A lie-in. Maybe a tea. That is not what happened.

The morning kicked off with PR #157 — the second wave of the LSP core extraction. More of the language server's private logic migrated into the shared `@satsuma/core` library. CST text helpers, workspace index field extraction, viz-model spread resolution, classification — anything that had been quietly living in the LSP and needed to live somewhere the CLI and other consumers can reach. Twenty-six tests. Done.

Then: **forty-eight bugs.** PR #165 was filed as "exploratory bug hunt" and honestly that's the least dramatic way to describe it. Someone sat down with the CLI and systematically tried things that should work and things that shouldn't and wrote a ticket for everything that behaved wrong. Forty-eight tickets. In one session. Two ADRs (ADR-022 and ADR-023) came out of it. Two PRDs for work that needs doing. I genuinely respect the commitment to writing things down instead of just quietly fixing them.

ADR-022 landed as an implementation the same day (#167). The finding was that the CLI was accepting directory paths as arguments — so you could run `satsuma validate my-pipeline/` and it would try to validate everything in that folder. That sounds convenient but creates a mess: the language has explicit imports, so validation should start from a specific entry-point file and follow the import graph, not do a flat directory scan. The fix is straightforward: the CLI now rejects directory arguments and requires a `.stm` file entry point. If you give it a folder, it tells you what it wants instead. ADR-022 documents the decision.

Three more bug fixes shipped while all of this was happening: the formatter was swallowing comments that appeared between an opening `{` and the first body node — PR #166 fixed that. The parser was silently accepting empty backtick identifiers like ` `` ` — PR #168 made those a proper parse error. PR #169 standardised JSON output shape across seven CLI commands that had each been doing their own thing.

And yes, six dependabot bumps also landed: GitHub Actions updated to their latest major versions (checkout v6, setup-node v6, upload-artifact v7, configure-pages v6, test-reporter v3) and typescript-eslint to 8.58.0. One lint fix to remove the now-empty `features/` directory from the ruff check paths, because you cannot lint a directory that does not exist.

Two days in a row. The project went from "shared library extracted" to "forty-eight bugs filed, CLI contract tightened, and JSON normalised across the board." Whatever happens next, the codebase is considerably more honest about what it does and doesn't do than it was on Saturday.
