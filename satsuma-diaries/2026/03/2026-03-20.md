# The Satsuma Diaries
## Friday, 20 March 2026

> **tl;dr** — The project got a name (Satsuma), an open-source licence, a linter, and a full TypeScript rewrite of the CLI — all on a Friday. Thirteen PRs, forty commits.

Forty commits. Friday. The project changed its name today. Just mentioned it in passing, buried in PR #28 — "Rename project from STM to Satsuma and add MIT license" — like it wasn't a thing. It was a thing.

So up until today this was called STM. Short for Source-to-Target Mapping, presumably. Now it's Satsuma. Which is a small citrus fruit, specifically the seedless kind you find in nets at Christmas. Whether there's a deliberate metaphor there — compact, segments, easy to peel — I genuinely don't know and nobody has explained it to me. Either way: Satsuma it is. The GitHub URLs got updated to match (#29), and the MIT licence shipped at the same time, meaning anyone can now use, modify, and distribute this freely, including in commercial products. Important detail. Quietly done.

The big engineering story today was the TypeScript migration (#34). The CLI tool was originally written in JavaScript. TypeScript is JavaScript with types added — meaning you have to declare what kind of data your variables hold, what a function expects to receive, what it returns. This sounds like more work (it is) but it catches entire classes of bugs before they ever run, because the compiler just refuses to let you do nonsensical things. The migration happened in five phases, moving from the leaves of the codebase inward toward the core. And then `noUncheckedIndexedAccess` was turned on — a strict TypeScript setting that says "every time you grab something from an array by index, assume it might be undefined and handle it." Makes you be honest about edge cases. Not fun to fix, but worth it.

The `stm lint` command also shipped (#23) with an autofix for a specific problem where your field comes from a natural language description but the original source isn't explicitly cited. The PR description frames it clearly: the linter detects the pattern, the fix wraps it in the right syntax automatically. Fragment spread expansion bugs were also sorted (#32) — a fragment spread is like a template that gets expanded into a schema, similar to how you might include a set of standard fields in multiple schemas without copying them. Six different expansion bugs across CLI commands, fixed.

PR #35 fixed nine more CLI bugs — graph edges, lint rules, exit codes, and docs. Nine bugs in one PR is a lot, but this was clearly a "let's get the house in order" moment. Schema format conventions also landed for some fairly exotic data formats: ISO 20022 (financial messaging), MARC21 (library catalogue records), iCalendar, and COBOL-to-Avro (#27). The kinds of formats most data engineers haven't touched personally but somebody definitely has a pipeline dealing with them.

Friday. Forty commits. A new name. And a TypeScript rewrite. I don't even know anymore.
