# The Satsuma Diaries
## Tuesday, 24 March 2026

> **tl;dr** — The formatter shipped, the website launched, the VS Code parser went cross-platform, security scanning was added, thirty dependency updates merged, and v0.1.0 was released. Forty-one PRs, ninety-four commits. I'm fine.

Ninety-four commits. Tuesday. I'm just going to write it all down and try to stay calm.

The formatter shipped (#91). `satsuma fmt` — a command that takes any Satsuma file and rewrites it in the canonical style, consistent indentation, consistent spacing, consistent everything. The PR mentions eighty-one tests. The formatter has eighty-one tests. There's a `--check` mode for CI so the pipeline will fail if someone pushes an unformatted file, and the VS Code LSP server got a DocumentFormattingProvider which means you can just press format in the editor and it runs. Clean. Proper. Done.

The GitHub Pages site launched (#82). Satsuma now has a website. It has a hero section, a feature cards section, a quick-start guide, a FAQ, and a landing page with a working code example. The PR mentions that it uses Tailwind CSS with the vendor file included locally because the security check requires SRI — Subresource Integrity, a browser security feature that ensures scripts haven't been tampered with. Good detail. The VS Code extension is linked on the site as step two in the install flow. This is now a public-facing project with a homepage. That's not nothing.

v0.1.0 was also prepped and released (#80, #81). The first versioned release. CI was wired to automatically attach build artifacts to tagged releases, and the site was updated to link to the actual v0.1.0 install files rather than placeholders.

The VS Code extension's LSP server was migrated to WASM for the tree-sitter parser (#87). WASM — Web Assembly — means the parser runs in a sandboxed, portable environment rather than depending on a native binary compiled for a specific platform. Practically: it works on Mac, Linux, and Windows without rebuilding the parser for each one. That motivation becomes obvious when you look at the CI matrix: Darwin, Linux, Windows, all in the release targets.

Semgrep and Dependabot were added for security (#57). Semgrep is a static analysis tool that scans code for security issues. Dependabot is GitHub's automated dependency-update bot. A CodeQL finding was also patched (#58) — client-side cross-site scripting in the GitHub Pages site. And then Dependabot got to work: about twenty dependency bumps merged on the same day (#59–#76) — ESLint 10, TypeScript 6, various GitHub Actions bumped to v5/v6. The TypeScript 6 migration required config updates for compatibility, which is the kind of invisible-but-real work that takes longer than it looks.

The Excel-to-Satsuma skill was rearchitected as a proper Agent Skill (#92) — the conversion prompt moved from `useful-prompts/` into the formal skills directory with a dedicated spec, plus a new `excel_tool.py` with five subcommands for interrogating spreadsheets programmatically. And a Satsuma-to-Excel lite prompt was written alongside it, so you can go in both directions using a web AI tool.

VS Code navigation improvements and TODO diagnostics landed (#90). Convention docs, tutorials, and canonical examples for common modelling patterns shipped (#93). The EDI convention guide covered EDIFACT, ODETTE, and TRADACOMS (#94). The MARC21 convention gaps were closed (#95). And the site got a callout elevated for the extensible metadata superpower (#96). I counted. That's a lot of separate things on a Tuesday.

Ninety-four commits. I'm going to go lie down.
