# The Satsuma Diaries
## Tuesday, 24 March 2026

Ninety-four commits. Tuesday. I'm just going to write it all down and try to stay calm.

The formatter shipped. `satsuma fmt` — a command that takes any Satsuma file and rewrites it in the canonical style, consistent indentation, consistent spacing, consistent everything. Eighty-one tests. The formatter has eighty-one tests. There's also a `--check` mode for CI so the pipeline will fail if someone pushes an unformatted file, and the VS Code LSP server got a DocumentFormattingProvider which means you can just press format in the editor and the formatter runs. Clean. Proper. Done.

The GitHub Pages site launched. Satsuma now has a website. It has a hero section, a feature cards section, a quick-start guide, a FAQ, a landing page with a code example, and it uses Tailwind CSS with the vendor file included locally because the security check requires SRI (Subresource Integrity — a browser security feature that ensures scripts haven't been tampered with). The VS Code extension is linked on the site as step two in the install flow. This is now a public-facing project with a homepage.

The Excel-to-Satsuma skill was also rearchitected as a standard Agent Skill. Skills are a specific format for packaging up a task for an AI agent to execute — they have a defined spec, they're self-contained, and they can be invoked by name. The conversion prompt was moved from the useful-prompts folder into the proper skills directory in the correct format, including a new `excel_tool.py` with five subcommands for interrogating spreadsheets programmatically. And a Satsuma-to-Excel lite prompt was written — so now you can go in both directions using a web AI tool.

The VS Code extension's LSP server was also migrated to use WASM (Web Assembly) for the tree-sitter parser. This means the parser runs in a sandboxed, portable environment rather than depending on a native binary compiled for a specific platform. Practically: it works on Mac, Linux, and Windows without needing to rebuild the parser for each one. The motivation becomes obvious when you look at the CI matrix: Darwin, Linux, Windows, all in the release targets.

Semgrep and Dependabot were added for security — Semgrep is a static analysis tool that scans code for security issues, Dependabot is GitHub's automated dependency-update bot. A CodeQL SAST finding (Static Application Security Testing, basically automated vulnerability scanning) was also patched — client-side cross-site scripting in the GitHub Pages site.

There were dependency bumps too. ESLint 10, TypeScript 6, various GitHub Actions bumped to v5/v6. The TypeScript 6 migration required config updates for compatibility, which is the kind of invisible-but-real work that takes longer than it looks.

Ninety-four commits. Tuesday. I'm going to go lie down.
