Satsuma CLI

Deterministic structural extraction for Satsuma workspaces. Every command produces 100% correct results from the parse tree — no regex heuristics, no guessing.

The CLI extracts structural facts. It does not interpret natural language. Satsuma uses NL strings for intent that cannot be captured as deterministic pipelines — the CLI delivers them verbatim. The CLI is the toolkit. The agent is the runtime.

Workspace Extractors

Block-level extraction

Retrieve whole blocks or workspace-level summaries. These commands give agents and humans fast access to schemas, mappings, lineage, and metadata without reading raw .stm files.

summary

Workspace overview — all schemas, mappings, metrics, and counts at a glance.

$ satsuma summary examples/

schema

Full schema definition extracted from the parse tree, including fields, types, and metadata.

$ satsuma schema hub_customer

metric

Full metric definition from the parse tree, with formula and dimensions.

$ satsuma metric monthly_revenue

mapping

Full mapping with all arrows and transforms. Quote names that contain spaces.

$ satsuma mapping 'sfdc to hub_customer'

find --tag

Find all fields carrying a specific metadata tag. Essential for PII audits.

$ satsuma find --tag pii

lineage

Schema-level graph traversal. Trace data flow forward or backward through the workspace.

$ satsuma lineage --from loyalty_sfdc

where-used

All references to a schema, fragment, or transform across the workspace.

$ satsuma where-used hub_product

warnings

All //! and //? comments across the workspace.

$ satsuma warnings examples/

context

Keyword-ranked block extraction. Returns the most relevant blocks for a query (heuristic).

$ satsuma context "customer mapping"
Structural Primitives

Fine-grained extraction

Slice below block level to get specific arrows, NL content, metadata, or field lists. These primitives are the building blocks agents compose into higher-level workflows.

arrows

All arrows involving a field, with transform classification: structural, nl, mixed, or none.

$ satsuma arrows loyalty_sfdc.LoyaltyTier

nl

NL content — notes, transforms, and comments — extracted verbatim from a scope.

$ satsuma nl 'demographics to mart'

meta

Metadata entries for a block or field — tags, constraints, and annotations.

$ satsuma meta loyalty_sfdc.Email

fields

Field list with types and metadata for a schema. Supports --unmapped-by filtering.

$ satsuma fields sat_customer_demographics

match-fields

Normalized name comparison between two schemas. Finds likely field matches for mapping drafts.

$ satsuma match-fields --source loyalty_sfdc --target sat_customer_demographics
Analysis

Format, validate, lint, and diff

One canonical style, structural correctness checks, policy conventions, and workspace comparison.

fmt

Opinionated, zero-config formatter. One canonical style. Also available as VS Code Format Document.

$ satsuma fmt --check examples/
0 file(s) would be reformatted

validate

Parse errors and semantic reference checks. Answers: “Is this workspace well-formed?”

$ satsuma validate examples/
valid — 0 errors, 0 warnings

lint

Policy and convention checks. Answers: “Does this workspace follow best practices?”

Supports --fix for safe, deterministic autofix. Use --rules to list available rules.

$ satsuma lint --fix --json
warning hidden-source-in-nl fixable
warning unresolved-nl-ref
error duplicate-definition

diff

Structural comparison of two workspace snapshots. Shows added, removed, and changed definitions.

$ satsuma diff v1/ v2/ --json
Graph

Full workspace topology

Export the complete semantic graph — nodes, edges, and field-level data flow — in a single call. Use flags to control scope and payload size.

--json Full graph with all nodes and edges
--compact Schema-level adjacency list
--schema-only Topology only, omit field-level edges
--namespace <ns> Filter to a single namespace
--no-nl Strip NL text from edges for smaller payloads

How agents use the CLI

The CLI is a set of structural primitives the agent composes into workflows. The agent — not the CLI — performs higher-level analysis.

Impact Analysis

# 1. Get arrows from the source field
$ satsuma arrows loyalty_sfdc.LoyaltyTier --as-source --json

# 2. Follow the chain
$ satsuma arrows sat_customer_demographics.loyalty_tier --as-source --json

# 3. When a hop is classified [nl], read the NL
$ satsuma nl mart_customer_360.loyalty_tier

PII Audit

# 1. Find all PII-tagged fields
$ satsuma find --tag pii --json

# 2. For each, trace outbound arrows
$ satsuma arrows loyalty_sfdc.Email --as-source --json

# 3. Agent reads NL to judge if PII survives
$ satsuma nl mart_customer_360.email

Coverage Assessment

# 1. Unmapped fields from a specific mapping
$ satsuma fields mart_customer_360 --unmapped-by 'demographics to mart' --json

# 2. Agent intersects results across mappings
$ satsuma fields mart_customer_360 --unmapped-by 'online to mart' --json

Reviewing a Change

# 1. Structural diff between branches
$ satsuma diff main-branch/ feature-branch/ --json

# 2. Check downstream arrows for changed fields
$ satsuma arrows changed_schema.changed_field --as-source --json

Transform classification

Every arrow the CLI returns carries a classification derived from CST node types. No string content is examined — the first four are mechanical CST checks.

Classification Meaning
structural Deterministic pipeline — fully specified by the syntax
nl Natural-language string — extracted verbatim for agent interpretation
mixed Both pipeline steps and NL strings
none No transform body (bare src -> tgt)
nl-derived Implicit arrow inferred from a backtick NL reference

Design boundaries

Does not interpret NL

Transform strings, notes, and comments are extracted verbatim. The CLI never assesses whether an NL transform is correct or complete.

Does not compose analysis workflows

There are no impact, coverage, or audit commands. These are agent workflows built from primitives.

Does not call language models

The CLI is deterministic, fast, and reproducible. Same input, same output, every time.

Does not accept NL queries

Commands take explicit structural arguments. The agent decides which commands to call based on the user's question.
