# schema: fields after RECORD/LIST blocks are silently dropped

Fields appearing after a RECORD or LIST block's closing brace in a schema are silently dropped from both text and JSON output.

## Reproduction

Given this .stm file:

```stm
schema with_record {
  id       INTEGER  (pk)
  address  RECORD {
    street   STRING(200)
    city     STRING(100)
    zip      STRING(10)
  }
  tags     LIST[STRING]
}
```

Run: `satsuma schema with_record <file> --json`

Expected: fields array contains ["id", "address", "street", "city", "zip", "tags"]
Actual: fields array contains ["id", "address", "street", "city", "zip"] — the `tags` field after the RECORD block is missing.

Same issue with nested records — any field after a nested RECORD's closing brace is dropped.

Additionally, RECORD sub-fields are flattened to the same level as top-level fields with no indication of nesting in the fields array.

## Root cause

Likely in extract.ts — the CST traversal for schema fields stops or resets after encountering a RECORD/LIST block's closing brace instead of continuing to the next sibling.

## Impact

High — fields silently disappear from schema output, causing downstream tools to work with incomplete data.

**2026-03-22T08:15:00Z**

Cause: The ticket's reproduction used invalid syntax (`address RECORD { ... }`) instead of correct Satsuma v2 syntax (`record address { ... }`). With correct syntax, the parser correctly produces `record_block` nodes and `extractFieldTree` correctly captures all sibling fields after nested blocks.
Fix: No code change needed for this specific bug. The grammar fix for sl-9uh0 (token.immediate dots in path rules) was the real parser fix in this epic. The field extraction logic in extract.ts was already correct.
