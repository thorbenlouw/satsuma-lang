# graph --namespace: schema_edges and edges reference nodes not in the filtered nodes array

When using `graph --namespace <ns>`, the result includes schema_edges and field-level edges that reference nodes (mappings and schemas) from other namespaces that are not present in the `nodes` array. This creates orphan edges pointing to/from nonexistent node IDs in the filtered view.

**Commands to reproduce:**
```bash
npx satsuma graph /tmp/satsuma-test-lineage-graph/namespaces.stm --json --namespace crm
```

**Expected:** Either (a) include cross-namespace mapping/schema nodes that touch the filtered namespace in the nodes array, OR (b) exclude the edges referencing those nodes.
**Actual:** schema_edges includes `crm::customers -> crm_to_warehouse [source]` but `crm_to_warehouse` is NOT in the nodes array. Field-level edges include `crm::customers -> warehouse::dim_customer` but `warehouse::dim_customer` is not in nodes.

Same issue for `--namespace warehouse`: `crm_to_warehouse -> warehouse::dim_customer [target]` appears in schema_edges but `crm_to_warehouse` is not in nodes.

**Fixture:** /tmp/satsuma-test-lineage-graph/namespaces.stm


## Notes

**2026-04-02T00:00:00Z**

Cause: `buildSchemaEdges` emitted edges referencing cross-namespace nodes (sources, targets, bridging mappings) but `buildWorkspaceGraph` only added in-namespace nodes to the `nodes` array — leaving edge endpoints without backing node entries.
Fix: Added a post-processing step in `buildWorkspaceGraph` after `buildSchemaEdges` that backfills any node referenced as an edge endpoint but not yet in `nodes`, looking it up in `index.schemas`, `index.mappings`, or `index.metrics` (`commands/graph-builder.ts`).
