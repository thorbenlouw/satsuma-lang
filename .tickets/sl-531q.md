# arrows: JSON source field uses lookup schema instead of actual source schema

When looking up arrows via a target field, the JSON output's 'source' property incorrectly uses the lookup schema (the target schema) instead of the actual source schema.

What I did:
  satsuma arrows target_sys.is_active --json /tmp/satsuma-test-arrows/all-arrows.stm
  satsuma arrows snowflake_opps.opp_key --json examples/sfdc_to_snowflake.stm

What I expected:
  { "source": "source_sys.status", "target": "target_sys.is_active", ... }
  { "source": "sfdc_opportunity.Id", "target": "snowflake_opps.opp_key", ... }

What actually happened:
  { "source": "target_sys.status", "target": "target_sys.is_active", ... }
  { "source": "snowflake_opps.Id", "target": "snowflake_opps.opp_key", ... }

The source field name is correct ('status', 'Id') but it's qualified with the wrong schema — it uses whatever schema was passed in the CLI argument rather than the schema where the source field actually lives.

When looked up from the source side, the attribution is correct:
  satsuma arrows source_sys.status --json => { "source": "source_sys.status", ... } (correct)

This is critical for agent workflows like impact analysis where arrows are chased downstream — the wrong source schema breaks the chain.

The text output is unaffected (it shows bare field names without schema qualification).

Reproducer file: /tmp/satsuma-test-arrows/all-arrows.stm


## Notes

**2026-03-22T02:00:00Z**

Cause: Arrows JSON used the schema passed to lookup instead of the arrow's actual source schema from the mapping.
Fix: Use actual source schema from mapping for JSON source field (commit 56f17ed).
