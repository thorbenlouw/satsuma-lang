# arrows: nl-derived arrows not returned when querying by target field

When an NL transform body contains an @ref (e.g. '"Sum @source8.amount grouped by order"'), the CLI correctly creates an nl-derived arrow from source8.amount to target8.total. However, this nl-derived arrow is only discoverable when querying the SOURCE field — it does NOT appear when querying the TARGET field.

Repro:
  # Fixture: -> total { "Sum @source8.amount grouped by order" }

  satsuma arrows source8.amount --json /tmp/satsuma-test-arrows-fl/08-nl-ref.stm
  => returns 1 arrow: nl-derived, source=source8.amount, target=target8.total  ✓

  satsuma arrows target8.total --json /tmp/satsuma-test-arrows-fl/08-nl-ref.stm
  => returns 1 arrow: the declared derived nl arrow (source=null) only
  => MISSING: the nl-derived arrow from source8.amount  ✗

  satsuma arrows target8.total --as-target --json /tmp/satsuma-test-arrows-fl/08-nl-ref.stm
  => same — only the declared arrow, not the nl-derived one  ✗

This is inconsistent with field-lineage, which correctly shows source8.amount in the upstream of target8.total:
  satsuma field-lineage target8.total --upstream --json
  => upstream: [{ field: '::source8.amount', classification: 'nl-derived' }]  ✓

Expected: querying arrows for target8.total (with or without --as-target) should include the nl-derived arrow alongside the declared derived arrow.


## Notes

**2026-04-01T07:18:21Z**

The original expectation (nl-derived arrows visible when querying by target) conflicts with the established BDD contract. The fix instead corrects the alreadyDeclared check so that when a declared arrow has source=null (computed), the nl-derived arrow is not suppressed for source-side queries. Target-side nl-derived discovery is handled by field-lineage, not arrows.
